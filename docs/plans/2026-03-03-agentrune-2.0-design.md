# AgentRune 2.0 — Differentiation Design

## Product Definition

**AgentRune** is the human command center for Web 4 — the agent-driven internet era.

In a world where AI agents do the work, humans don't need to stare at terminals. They need to express intent, review decisions, and track progress. AgentRune lets you command any AI agent from your phone, the human-friendly way.

**Positioning:** Agent-agnostic mobile command center. Not a terminal emulator — a Mission Control for AI agents.

**Core workflow:** Open → Pick project → Pick agent → Mission Control (task board) → Direct via cards, not terminal.

**Ecosystem:**
- **AgentLore** (infrastructure) — defines and manages agent capabilities
- **AgentRune** (human entry point) — lets humans operate these agents
- **AI Agents** (Claude Code, OpenClaw, Codex, Aider...) — the commanded entities

**Brand narrative:** *Web 4 is agent-driven. AgentLore builds their world. AgentRune is your language.*

## Competitive Landscape

| Product | What it is | AgentRune's advantage |
|---------|-----------|----------------------|
| Claude Remote Control | Official mobile Claude Code control | Only supports Claude Code; pure terminal view |
| Happy Coder | Open-source mobile client for Claude + Codex | Pure terminal forwarding, no task abstraction |
| OpenClaw (via messaging) | AI agent controlled through WhatsApp/Telegram | Chat interface is not task-oriented; no visual structure |
| Termius | SSH client with AI features | DevOps-focused, not AI coding |

**AgentRune's moat:** The only product that provides a structured task board UI across multiple AI agents. Every other product is either terminal-based or locked to a single agent.

**New core modules:**
1. **Parse Engine** — converts raw terminal output into structured AgentEvents
2. **Agent Registry** — per-agent parse rules, launch commands, capability declarations
3. **Event Store** — persists parsed event streams for board and history

## Parse Engine

### Event Model

```typescript
interface AgentEvent {
  id: string
  timestamp: number
  type: "file_edit" | "file_create" | "file_delete"
      | "command_run" | "test_result" | "install_package"
      | "decision_request" | "error" | "info" | "session_summary"
  status: "in_progress" | "completed" | "failed" | "waiting"
  title: string           // "Editing server/auth.ts"
  detail?: string         // "+12 lines -3 lines"
  raw?: string            // corresponding raw terminal output
  decision?: {            // only for decision_request type
    options: { label: string; input: string; style: "primary" | "danger" | "default" }[]
  }
}
```

### Three-Layer Parse Strategy

```
┌─────────────────────────────────────────┐
│  Layer 3: Agent structured protocol     │
│  Most precise, requires agent support   │
│  e.g. Claude Code JSON streaming events │
├─────────────────────────────────────────┤
│  Layer 2: Agent-specific patterns       │
│  Regex/parser for specific agent output │
│  e.g. Codex "Editing file..." messages  │
├─────────────────────────────────────────┤
│  Layer 1: Universal terminal patterns   │
│  Works with any agent                   │
│  e.g. Y/n prompt, $ idle, error keyword │
└─────────────────────────────────────────┘
```

### Agent Adapter Interface

```typescript
interface AgentAdapter {
  id: string
  name: string
  icon: string
  launch: (settings: ProjectSettings) => string | null
  parse: (chunk: string, context: ParseContext) => AgentEvent[]
  detectIdle: (buffer: string) => boolean
  capabilities: string[]
}
```

### Agent Support Matrix

| Agent | Priority | Parse Layer | Detectable Events |
|-------|----------|-------------|-------------------|
| Claude Code | P0 | L2+L3 | Tool calls, file changes, test results, decisions, token usage |
| Codex CLI | P0 | L2 | File edits, command execution, approval requests |
| OpenClaw | P0 | L2 | Skill execution, task completion, errors |
| Terminal | P0 | L1 | Idle/running, Y/n prompts, error output |
| Aider | P1 | L2 | File edits, git commits, diff output |
| Cline / Continue | P2 | L1→L2 | Community-driven |

### OpenClaw Integration

Two modes:

**Mode A: CLI Agent launch**
- AgentRune starts OpenClaw via node-pty: `openclaw chat --provider claude`
- Same integration pattern as Claude Code / Codex
- Parse Engine uses OpenClaw L2 adapter

**Mode B: Gateway connection**
- Connects to already-running OpenClaw Gateway (localhost:3000)
- AgentRune acts as a better frontend than WhatsApp/Telegram
- Requires Gateway Connector module

### Data Flow

```
terminal stdout → buffer accumulation
                     ↓
              Agent Adapter.parse()
                     ↓
              AgentEvent[] generated
                     ↓
          ┌──────────┴──────────┐
          ↓                     ↓
    Event Store              WebSocket → Mission Control UI
    (server-side persist)    (real-time push to phone)
```

### Card Types

| Type | Trigger | Content | Interaction |
|------|---------|---------|-------------|
| ⚠️ Decision | Agent awaits human input | Question + option buttons | Tap button to respond |
| 🔄 In Progress | Agent executing an operation | Operation description + progress | Tap to expand terminal |
| ✅ Completed | Operation done | Result summary | Tap for detail |
| ❌ Error | Operation failed | Error message + suggestion | Tap for stack trace |
| 📊 Session Summary | Session ends or goes idle | Stats: files, lines, tests | — |

### Status Indicator

| State | Visual | Meaning |
|-------|--------|---------|
| 🟢 Working | Green pulsing dot | Agent working autonomously |
| 🟡 Waiting | Yellow flash + vibration | Agent blocked, needs your response |
| ⚪ Idle | Gray static | Agent finished, awaiting next instruction |

### Key Interactions

- **Decision cards always pinned to top** — Agent is waiting, highest priority
- **Tap any card** → expand to show terminal output snippet around that operation
- **Pull down past bottom** → full Terminal Detail view (escape hatch)
- **Long press card** → copy relevant info (filename, command, error)

## Session History

### Session Model

```typescript
interface Session {
  id: string
  projectId: string
  agentId: string
  startedAt: number
  endedAt?: number
  status: "active" | "completed" | "killed"
  events: AgentEvent[]
  summary?: {
    filesModified: number
    filesCreated: number
    linesAdded: number
    linesRemoved: number
    testsRun?: number
    testsPassed?: number
    decisionsAsked: number
    duration: number
  }
}
```

### Data Storage

| Data | Location | Format |
|------|----------|--------|
| Session events | Server: `.agentrune/sessions/` | JSON per session |
| Session index | Server: `.agentrune/sessions/index.json` | Summary list |
| User preferences | Client: `localStorage` | `agentrune_*` keys |
| Agent adapters | Server: `adapters/` | TypeScript modules |

File-based JSON storage — zero dependencies, simple, git-trackable.

## Design Language

### Glassmorphism Rules

**Critical principle:** Glass is transparent and neutral. Color is expressed through accents (left border, icons), never by tinting the glass itself. This prevents colored glass panels from interfering with each other when layered.

| Element | Specification |
|---------|--------------|
| **Card base (all cards)** | `backdrop-filter: blur(24px)` + `background: rgba(255,255,255,0.04)` + `border: 1px solid rgba(255,255,255,0.08)` — always neutral |
| **Decision card** | Neutral glass + `border-left: 4px solid rgba(250,204,21,0.6)` yellow |
| **Error card** | Neutral glass + `border-left: 4px solid rgba(248,113,113,0.6)` red |
| **Success card** | Neutral glass + `border-left: 4px solid rgba(74,222,128,0.6)` green |
| **In-progress card** | Neutral glass + `border-left: 4px solid rgba(96,165,250,0.6)` blue |
| **Bottom sheet** | `backdrop-filter: blur(32px)` + `background: rgba(255,255,255,0.06)` |
| **Nav bar** | `backdrop-filter: blur(16px)` + `background: rgba(255,255,255,0.03)` |

**Blur hierarchy:** Nav (16px) < Cards (24px) < Overlays (32px) — higher layers blur more.

## AgentLore Integration Phases

| Phase | Integration | Effect |
|-------|------------|--------|
| Phase 1 | Agent Registry syncs from AgentLore API | Agent definitions no longer hardcoded |
| Phase 2 | Project config pulled from AgentLore | No manual `projects.json` maintenance |
| Phase 3 | AgentLore Workflow support | Launch and monitor multi-step workflows from AgentRune |

### API Contract (Phase 1)

```
GET  /api/agents     → AgentAdapter[] definitions
GET  /api/projects   → Project[] configuration
POST /api/events     → Report AgentEvent back to AgentLore (bidirectional sync)
```
